3. 异常处理流程

解释：

  • 无参数的 throw 用于重新抛出当前捕获的异常。
  • 例如： 在 catch 块中捕获一个 int 类型的异常后，通过无参数的 throw 关键字将同样的异常对象重新抛出。

作用：

  • 重新抛出异常可以将异常传播到更高层的调用者，适用于某些情况下需要中途处理异常但无法完全解决的情景。

注意：

  • 无参数的 throw 只能在 catch 块中使用，否则会导致未定义行为。

如何应对多出口引发的处理碎片？

PPT中的这个问题是指异常处理的设计可以避免程序中由于多种错误出口导致的代码碎片化问题。

背景：

  • 在传统的错误处理机制中，通常通过返回错误代码或设置全局变量来传递错误信息，这种方法可能导致代码复杂且难以维护。
  • 异常处理通过将错误的处理逻辑集中到 catch 块中，减少了多出口的代码碎片。

总结

  1. 无参数 throw：

    • 用于在 catch 块中重新抛出异常，通常用于将异常传递给更高层的异常处理器。

  2. catch(...)：

    • 捕获所有类型的异常，通常作为程序最后的安全保障机制。

  3. 对程序验证特性的支持：

    • 通过模板断言函数（Assert），结合异常处理机制实现程序的运行时验证，尤其适合调试模式。

  4. 实现特点：

    • 异常处理机制不影响对象布局，并在异常抛出时自动调用对象的析构函数进行资源释放，确保程序状态的一致性。

  5. 多出口问题：

    • 异常处理机制有效解决了传统错误处理方式中多出口导致的代码碎片问题，使代码更易于维护和扩展。

核心概念：重定向

在C++中，cin 和 cout 默认使用标准输入（键盘）和标准输出（控制台）作为数据流的来源和目标。通过流缓冲区（streambuf），可以将这些流重定向到文件或其他流对象，实现输入输出的灵活控制。

代码解析

1. 重定向 cin 到文件 in.txt

  • ifstream in("in.txt")：
    • 创建一个输入文件流 in，从文件 in.txt 中读取内容。
  • cin.rdbuf()：
    • 获取当前 cin（标准输入流）的缓冲区指针，并保存到 cinbuf 中，便于后续恢复。
  • cin.rdbuf(in.rdbuf())：
    • 将 cin 的缓冲区设置为文件流 in 的缓冲区，这样后续通过 cin 的输入会从 in.txt 文件中读取，而不是从标准输入（键盘）读取。

2. 重定向 cout 到文件 out.txt

  • ofstream out("out.txt")：
    • 创建一个输出文件流 out，用于向文件 out.txt 写入内容。
  • cout.rdbuf()：
    • 获取当前 cout（标准输出流）的缓冲区指针，并保存到 coutbuf 中，便于后续恢复。
  • cout.rdbuf(out.rdbuf())：
    • 将 cout 的缓冲区设置为文件流 out 的缓冲区，这样后续通过 cout 的输出会写入到 out.txt 文件，而不是标准输出（控制台）。

3. 使用 cin 和 cout 进行文件输入和输出

  • cin >> word：
    • 因为 cin 已经被重定向到文件 in.txt，因此此处会从 in.txt 中读取一个单词并存储到变量 word 中。
  • cout << word << " "：
    • 因为 cout 已经被重定向到文件 out.txt，因此此处会将变量 word 的值写入文件 out.txt，并在后面加上一个空格。

4. 恢复 cin 和 cout 的原始缓冲区

  • cin.rdbuf(cinbuf)：
    • 将 cin 的缓冲区恢复为原始的标准输入缓冲区，之后 cin 的输入将来自键盘。
  • cout.rdbuf(coutbuf)：
    • 将 cout 的缓冲区恢复为原始的标准输出缓冲区，之后 cout 的输出将显示到控制台。

5. 使用恢复后的 cin 和 cout

  • cin >> word：
    • 此时 cin 已经恢复为标准输入流，因此从键盘读取输入。
  • cout << word：
    • 此时 cout 已经恢复为标准输出流，因此将数据输出到控制台。

总结

  1. 输入重定向：

    • 通过 cin.rdbuf() 将标准输入流 cin 的缓冲区替换为文件流的缓冲区，可以从文件中读取数据。

  2. 输出重定向：

    • 通过 cout.rdbuf() 将标准输出流 cout 的缓冲区替换为文件流的缓冲区，可以将数据写入到文件中。

  3. 恢复重定向：

    • 重定向完成后，可以通过恢复原始缓冲区指针（streambuf*）的方式，将流重新设置为标准输入和标准输出。

  4. 用途：

    • 输入输出重定向可以用于测试程序的输入输出、数据日志记录等场景，简化操作。

Virtualizing constructors

在C++中，构造函数不能声明为虚函数（virtual）。然而，在某些情况下，需要根据运行时的类型动态创建对象，而这正是“虚拟构造函数”的核心概念。

内容分析

1. Virtual constructor

概念：

  • Virtual function（虚函数）：

    • 虚函数是支持运行时多态的关键机制，用于动态调用派生类中重写的函数。
    • 虚函数的核心作用是通过基类指针或引用调用派生类的实现。

  • Constructor（构造函数）：

这张PPT的主题是“Never treat arrays polymorphically”（永远不要将数组以多态的方式对待）。它强调了C++数组在多态上下文中的行为问题，特别是子类数组被当作基类数组使用时可能出现的问题。

问题背景

代码片段描述了一个基本的二叉搜索树类（BST）及其派生类（BalancedBST），并尝试通过一个函数处理二叉搜索树的数组。

  • 这个函数接收一个 ostream 对象、一个 BST 类型的数组和数组大小 numElements。
  • 目标： 遍历数组中的每个元素并将其输出到流 s。

  • 创建了一个大小为 10 的 BalancedBST 数组（子类对象）。
  • 然后尝试将这个子类数组传递给 printBSTArray 函数，其中参数的类型是 const BST array[]（基类数组）。

问题分析

C++ 不支持对数组的多态处理。以下是问题的根源：

1. 数组退化为指针

  • 在函数调用中，数组作为参数会退化为指向数组第一个元素的指针。
  • 在这段代码中： bBSTArray 会退化为 BalancedBST* 类型的指针，但函数期望的是 const BST* 类型的指针。

2. 对象切割问题

  • 虽然 BalancedBST 继承自 BST，但数组中的每个元素在函数内部会被当作 BST 类型的对象处理。
  • 数组元素被传递时，会发生对象切割（Slicing），即 BalancedBST 对象的派生类部分（子类新增的成员变量和函数）会丢失，剩下的只有基类 BST 的部分。

3. 虚函数对输出的影响

  • 如果 BST 类中定义了虚函数（如虚拟的输出运算符 operator<<），可以避免对象切割导致的问题：
  • 如果没有虚函数，s << array[i] 会调用基类的 operator<< 实现，而不是子类的。

输出操作问题：

  • 如果 BST 类没有定义虚拟的 operator<<，输出的内容只会涉及基类部分的数据，无法正确处理派生类的扩展部分。
  • 即使定义了虚拟函数，整个数组的多态行为也依然可能存在问题。

1. 使用指针数组

总结

  1. 数组的多态性问题：

    • 在 C++ 中，数组无法实现真正的多态。
    • 子类数组被当作基类数组时，会发生对象切割。

  2. 避免对象切割的方案：

    • 使用基类指针数组。
    • 在基类中定义虚拟的 clone 函数，动态创建派生类对象。
    • 使用 STL 容器和智能指针（推荐）。

  3. 更安全的设计：

    • 尽量避免直接使用数组，改用标准容器（如 std::vector）。
    • 通过虚函数实现运行时的动态分派，确保派生类对象被正确处理。
