继承

1. 继承机制简介

PPT 中提到：

继承机制是基于目标代码的复用

什么是继承？

继承（Inheritance）是面向对象编程（OOP）的三大特性之一，它允许一个类（子类或派生类）从另一个类（基类或父类）中继承属性和方法。

  • 基类（Base Class）：提供基础功能和接口。
  • 派生类（Derived Class）：继承基类的属性和方法，并可以扩展或修改这些功能。

继承的目的

  1. 代码复用：
    • 基类中的代码可以被子类直接使用，避免重复编写。
    • 提高了代码的可维护性和扩展性。
  2. 层次化表示事物：
    • 通过继承，可以对事物进行分类。
    • 从基类到派生类逐层增加具体化的细节。
  3. 支持多态（Polymorphism）：
    • 派生类对象可以被视为基类对象，从而实现动态绑定（Dynamic Binding）。

    • 基类使用虚函数（virtual）定义接口，派生类实现具体逻辑。

1. protected 访问权限概述

在 C++ 中，protected 是一种访问控制修饰符，其特性如下：

  • 基类内部：protected 成员可以在基类内部访问。
  • 派生类：protected 成员可以在派生类中直接访问。
  • 类外部：protected 成员不能被类外部直接访问。

3. 友元与访问权限

友元函数的特性

  • 友元函数被某个类声明为友元后，可以访问该类的所有成员，包括 private 和 protected。
  • 友元关系是单向的且非传递的：
    • 如果 A 是 B 的友元，B 并不会自动成为 A 的友元。

友元对继承成员的访问规则

  • 友元函数可以访问声明它的类的所有成员。
  • 如果一个派生类声明了某个函数为友元函数，该友元函数可以：
    • 访问派生类的私有成员。
    • 访问派生类从基类继承的 protected 成员。
  • 友元函数不能访问基类中直接声明的 protected 成员，除非该基类也将其声明为友元。

4. 总结

  1. protected 的访问规则：
    • protected 成员可以被基类和派生类直接访问，但不能被类外部访问。
    • 派生类中的 protected 成员仍然可以被派生类的友元函数访问。
  2. 友元函数的权限：
    • 可以访问友元类的所有成员（包括 private 和 protected）。
    • 不能访问友元类以外的成员（即使是 protected 成员）。
  3. 错误示例分析：
    • clobber(Base &b) 试图访问 Base 的 protected 成员，但它不是 Base 的友元函数，因此不允许。

1. 派生类对象的初始化

描述

  • 当创建派生类对象时，对象的初始化过程由 基类 和 派生类 共同完成。
  • 基类的构造函数会首先执行，然后是派生类的构造函数。

原因

  • 基类的成员是派生类对象的一部分，因此在派生类的构造函数运行之前，必须先初始化基类部分。

2. 构造函数的执行顺序

顺序

  • 基类的构造函数：先执行基类的构造函数，用于初始化基类的成员。
  • 派生类对象成员类的构造函数：如果派生类中包含对象成员（即类类型成员），会依次调用这些成员的构造函数。
  • 派生类的构造函数：最后执行派生类的构造函数，用于初始化派生类的特有成员。

3. 析构函数的执行顺序

顺序

析构函数的执行顺序与构造函数相反：

  1. 派生类的析构函数：先释放派生类的资源。
  2. 派生类对象成员类的析构函数：接着释放派生类中对象成员的资源。
  3. 基类的析构函数：最后调用基类的析构函数，释放基类的资源。

原因

  • 释放资源的顺序必须与构造顺序相反，以确保派生类中使用的基类资源先被正确释放。

1. using A::A 是什么？

using A::A 是 C++11 引入的一种新特性，用于继承基类的构造函数。这意味着，当派生类声明 using A::A 后，基类的构造函数会被引入到派生类中，可以直接用于派生类对象的构造。

换句话说：

  • 之前的写法（C++11 之前）：
    • 派生类必须手动定义构造函数，即使其逻辑和基类的构造函数完全相同。
  • C++11 后的改进：
    • 通过 using A::A，可以直接“继承”基类的构造函数，不需要重复书写。

2. 为什么需要继承基类的构造函数？

如果派生类没有特殊的初始化逻辑，而仅仅是想沿用基类的构造函数逻辑，继承构造函数可以减少代码重复，简化开发。

3. 继承构造函数的工作原理

继承构造函数会将基类的构造函数自动添加到派生类的构造函数集中，但有以下限制和特点：

  1. 构造函数本质上是“复制”到派生类中：
    • 派生类对象初始化时，基类的构造函数会像“被直接复制到派生类”一样被调用。
  2. 可以重写继承的构造函数：
    • 如果派生类需要对基类的某些构造函数添加新的逻辑，可以在派生类中重新定义这些构造函数，覆盖继承的构造函数。
  3. 不会继承默认构造函数：
    • 如果基类没有默认构造函数，派生类仍需要显式定义其自己的默认构造函数。

虚函数

1. 类型相容性概念

  • 类、类型：
    • C++ 中的对象可以分为基类（Base Class）和派生类（Derived Class），派生类继承了基类的特性。
    • 类型相容性主要指：一个派生类对象能否赋值或转换为基类类型，以及如何在运行时或编译时进行转换。
  • 类型相容性分为两种：
    • 赋值相容性：
      • 基类的指针（或引用）可以指向派生类的对象。
      • 但派生类的指针（或引用）不能直接指向基类的对象，必须进行显式类型转换。
    • 赋值相容时的行为：
      • 当基类指针指向派生类对象时，通过基类指针只能访问基类的成员，而派生类的独有成员将“不可见”。
      • 如果基类中的成员是虚函数，派生类的重写版本会在运行时被调用（多态的基础）。

6. 总结：派生类到基类的类型相容性

  1. 对象赋值：
    • 派生类对象赋值给基类对象时，合法，但会触发“对象切割”，丢失派生类的成员。
  2. 指针赋值（向上转型）：
    • 基类指针可以指向派生类对象，合法，但通过基类指针只能访问基类的成员。
  3. 引用赋值（向上转型）：
    • 基类引用可以绑定到派生类对象，合法，但只能通过基类引用访问基类的成员。
  4. 虚函数：
    • 如果基类成员是虚函数，通过基类指针或引用调用时，运行时会动态绑定到派生类的实现（多态）。

代码示例

    • 派生类对象不能直接接收基类对象赋值。
    • 原因是派生类需要扩展的部分无法从基类的部分推导出来。

代码示例

    • 使用基类的引用或指针，只能访问基类部分的成员。
  • B &r_b = a; 和 B *p_b = &a;
    • 向下转型（Downcasting）： 派生类的引用和指针不能绑定基类对象。这是非法的，因为基类对象不包含派生类的扩展部分。

    • 基类的引用或指针可以绑定派生类对象，但反之不成立。

7. 总结

  • 向上转型（Upcasting）是允许的，即基类的引用或指针可以绑定派生类对象。
  • 向下转型（Downcasting）是非法的，即派生类的引用或指针不能绑定基类对象。
  • 非虚函数在调用时是静态绑定的，调用的是对象的静态类型对应的函数。
  • 函数参数传递时，基类的引用或指针可以接收派生类对象，并根据虚函数机制决定调用哪一版本的成员函数。

1. 前期绑定（Early Binding）

定义：在编译时刻确定函数调用的具体实现，绑定到对象的静态类型。

  • 绑定时机：编译阶段。

  • 依据：静态类型（Static Type）。即对象在定义时的类型，而不是运行时对象的实际类型。

  • 特点：

    • 效率高：函数调用直接通过对象的静态类型进行，无需动态查表，速度较快。
    • 灵活性差：对象的实际类型可能与静态类型不一致，可能导致多态性丧失。

  • 典型场景：非虚函数。

    • 例如：

      • 尽管 p 指向一个 B 对象，但调用的是 A::func()，因为是基于静态类型 A 进行的绑定。

2. 后期绑定（Late Binding）

定义：在运行时刻根据对象的实际类型确定函数调用的具体实现。

  • 绑定时机：运行阶段。

  • 依据：动态类型（Dynamic Type）。即对象在运行时的实际类型，而不是编译时的静态类型。

  • 特点：

    • 灵活性高：可以实现多态，动态调用不同类型的函数。
    • 效率低：需要通过虚函数表（vtable）查找函数地址，会有一定的运行时开销。

  • 典型场景：虚函数（virtual）。

    • 例如：

      • 虽然 p 是 A* 类型，但实际指向的是一个 B 对象，因此调用 B::func()。

3. 注意效率

PPT指出：

  • 默认前期绑定：C++ 默认对非虚函数进行前期绑定，原因是效率更高。
  • 后期绑定需要显式声明 virtual：
    • 只有声明为虚函数的成员函数，才能实现后期绑定。
    • 虚函数的调用需要依赖虚函数表（vtable）和虚函数指针（vptr），因此效率稍低。

2. 虚函数的限制

（1）只有类的成员函数才能是虚函数

  • 非成员函数（如全局函数或普通非成员函数）不能是虚函数。

（2）静态成员函数不能是虚函数

  • 原因：静态成员函数与具体的对象无关，它们属于整个类，而虚函数依赖于对象的动态类型来实现动态绑定。

（3）内联成员函数不能是虚函数

  • 内联函数是建议编译器将其替换为函数体代码，从而优化性能。而虚函数需要通过虚函数表（vtable）动态查找，无法做到内联化。

（4）构造函数不能是虚函数

  • 构造函数在创建对象时被调用，而虚函数依赖于对象的动态类型。由于构造函数在对象创建过程中调用，动态类型尚未确定，因此构造函数不能是虚函数。

（5）析构函数可以是虚函数

  • 析构函数被设计为虚函数的原因在于，当通过基类指针删除派生类对象时，确保调用派生类的析构函数以释放派生类资源。

1. 背景知识：后期绑定（Late Binding）

  • 后期绑定的定义：在运行时决定调用哪一个函数，而不是在编译时。
  • 在C++中，如果类中包含虚函数（virtual修饰），就会创建一个虚函数表（vtable）。
  • 每个对象内部会有一个隐藏的指针（vptr），指向它对应的虚函数表。

2. 虚函数表的实现机制

虚函数表的工作原理如下：

  1. 虚函数表（vtable）：
    • 每个包含虚函数的类都有一张虚函数表。
    • 表中存储了该类的所有虚函数的地址（函数指针）。
    • 如果派生类重写了基类的虚函数，虚函数表中对应的地址会被派生类函数覆盖。
  2. 虚指针（vptr）：
    • 每个对象都有一个隐藏的指针，称为虚指针（vptr）。
    • vptr指向该对象所属类的虚函数表（vtable）。
    • 调用虚函数时，通过vptr访问虚函数表，从而实现动态绑定。

2. override 和 final 的作用

override

  • 定义：
    • 明确指定派生类的函数是用来重写基类的虚函数。
    • 编译器会检查函数签名是否和基类虚函数完全一致。
  • 作用：
    • 提高代码可读性和安全性，避免因函数签名不匹配导致的错误。
    • 避免无意中创建了一个新的函数（不是重写基类的函数）。

final

  • 定义：
    • 禁止派生类进一步重写某个虚函数。
    • 如果派生类尝试重写标记为final的虚函数，编译器报错。
  • 作用：
    • 提供明确的限制，防止虚函数被意外重写。
    • 常用于设计中，需要确保某些函数的行为在派生类中保持不变。

    • 如果基类成员函数不是虚函数，而派生类定义了同名函数，这将导致隐藏（隐藏基类的非虚函数），可能产生不符合预期的行为。
    • 推荐使用 virtual 明确成员函数的重载意图。

2. Rectangle 和 Square 的设计问题

问题背景

  • 从数学意义上，正方形是一种特殊的矩形（Square is a Rectangle）。
  • 然而，正方形的宽和高总是相等的，如果直接继承 Rectangle 并提供 setHeight 和 setWidth 的独立实现，会导致语义冲突。

代码分析

  • Square 重载了 setHeight 和 setWidth 并将其设为 private，从而限制了直接调用。
  • 但这种做法会导致多态行为的问题，因为基类的虚函数机制允许通过基类指针调用派生类的 setHeight 和 setWidth，从而破坏正方形的约束（宽等于高）。

Square::setHeight 是 private 的，能否调用？

  • 虽然 Square::setHeight 是 private 的，但这并不会阻止通过 基类指针 调用虚函数的过程。

  • 访问控制规则（public、protected、private）是在

    编译期检查的，而虚函数的解析是在运行时完成的。这是关键点：

    • 编译时：编译器只会检查 Rectangle 类的 setHeight 是否对调用方可见。
      • 因为 Rectangle::setHeight 是 public 的，所以编译器允许调用 p->setHeight(10)。
    • 运行时：由于 p 实际指向一个 Square 对象，虚函数机制会在运行时调用 Square 的 setHeight 方法，无视访问控制。

因此，即使 Square::setHeight 是 private 的，虚函数机制仍然会在运行时调用它。

为什么会导致 assert 失败？

  • 由于 Square::setHeight 是 private 的，开发者的本意可能是希望外部代码无法直接修改正方形的高度和宽度，以保持正方形的数学约束（宽等于高）。
  • 但通过基类指针调用虚函数时，Square::setHeight 被调用，而这个方法可能只修改高度，却没有同步修改宽度。
  • 结果就是：正方形的宽度和高度不再相等，破坏了正方形的语义。

private继承

  1. private继承的概念
  • private继承是C++中的一种继承方式。
  • 与public继承不同，private继承的目的并不是为了建立"is-a"（是一个）的关系，而是为了实现"implemented-in-term-of"（通过某个类的实现来实现另一个类的功能）。
  • 访问控制的特性：
    • 基类的public和protected成员，在派生类中都会变成private。
    • 派生类的外部无法直接访问基类的成员。
  • 设计目的：
    • 主要用于实现层面，而非设计层面，强调实现的内部逻辑，而不是在类型系统中建立父子关系。

(2) private继承的特性

  • private继承中，基类的成员在派生类中变为private，仅供派生类自身访问。
  • 外部代码无法直接访问基类的成员，也无法将派生类对象视为基类对象。
  1. private继承的用途

(1) 实现"implemented-in-term-of"

  • 通过private继承，派生类可以使用基类的protected成员或者覆盖其virtual函数，从而复用基类的实现。
  • 这种复用是“通过实现”而非“通过接口”。

(2) 隐藏基类的接口

  • private继承会隐藏基类的所有public和protected接口，从而防止派生类的用户直接访问这些接口。
  • 在设计上，强调“我实现了某些功能，但不希望外界知道我是通过继承某个类来实现的”。

(3) 强调实现层面的关系，而非类型层面的关系

  • 与public继承不同，private继承不会建立父子类的类型层次关系。
  • 在类型系统中，CStudent与CHumanBeing是完全不同的类型，不会因为继承而建立任何类型上的联系。

(1) 需要访问基类的protected成员

  • 如果需要访问基类的protected成员，但又不希望外界把派生类当成基类使用，可以采用private继承。

  1. 与组合的对比

在许多情况下，组合（has-a）比继承（特别是private继承）更合适：

  • 如果派生类只需要利用基类的实现，并不需要访问其protected成员，通常可以选择组合。
  • 组合的优势是更加灵活，能够明确表示语义上的关系：
    • 组合：表示“有一个”关系（has-a）。
    • 继承：表示“是一个”关系（is-a 或 implemented-in-term-of）。

  • 通过基类指针或引用调用虚函数

    • 虽然虚函数的动态绑定是运行时完成的，但缺省参数值是在编译时确定的。
    • 如果通过基类指针p_a调用f，缺省参数值使用的是基类中定义的值，而不是派生类中的值。
